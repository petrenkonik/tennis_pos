# ADR-0002: Rule-based анализ фаз (Подход A)

**Дата:** 2026-06-20
**Статус:** Принято

## Контекст

Нужно выбрать архитектуру оценки ошибок в подаче. Все рассмотренные подходы разделяют общий пайплайн:

```
Video → pose estimation → детекция фаз → анализ → фидбек
```

Различаются в том, **как анализируются фазы** и где живёт логика оценки ошибок.

## Рассмотренные подходы

### A. Rule-based анализ фаз ✅
Разбиваем подачу на фазы по детектируемым из pose-trajectory событиям, оценку ошибок делаем через **эвристические правила** на основе биомеханических эталонов.

### B. ML-классификатор ошибок
Та же пайплайн до фаз, но оценка — обученная модель (классификатор на векторе углов суставов по фазам).

### C. Гибрид — правила + LLM для объяснений
Эвристики детектят факты, LLM формулирует человекочитаемые объяснения.

## Решение

**Подход A — Rule-based анализ фаз.**

## Обоснование

1. **Прямо соответствует цели прототипа** — валидировать, что CV-пайплайн умеет разбивать подачу на фазы и находить ошибки. Правила делают «нахождение ошибок» прозрачным и проверяемым.
2. **Полностью в браузере** (см. ADR-0001) — никакого бэкенда.
3. **Объяснимость** — легко отлаживать («почему сказал, что ошибка?») и объяснять пользователю-любителю.
4. **Слои 2 и 3** (точные метрики, сравнение с эталоном) естественно ложатся на rule-based — это разные «views» над одними и теми же посчитанными углами.
5. **Не требует датасета** — в отличие от B, не нужно собирать и размечать сотни подач с метками ошибок.
6. **Переносим на будущий мобильный продукт** — архитектура фаз/событий переезжает без переписывания.

## Отклонённые альтернативы

### B. ML-классификатор — отклонён потому что:
- Нужен размеченный датасет подач с метками ошибок, которого нет
- Меньше объяснимости — сложно ответить «почему это ошибка»
- Требует где-то хостить/бандлить модель — противоречит «всё в браузере»
- Может быть пересмотрен, когда соберём датасет (future-work)

### C. LLM-объяснения — отклонён **на прототип**, оставлен как future-work:
- Добавляет серверный компонент (API-вызовы) → противоречит ADR-0001
- Стоимость за запрос + латентность
- Риск галлюцинаций в советах по технике (потенциально вредно для пользователя)
- Хорошая идея для улучшения человекочитаемого фидбека на более позднем этапе

## Последствия

### Положительные
- Детерминированный, объяснимый, легкий в отладке
- Не требует сбора данных
- Полностью клиентский

### Отрицательные / риски
- **Точность зависит от качества pose estimation** — mitigated использованием tolerance-зон в правилах
- **Правила нужно калибровать** вручную по эталонам — первая итерация будет приблизительной
- **Плохо ловит нюансы** неописанные правилами — приемлемо для прототипа; Layer 2 метрики дадут сырые данные для ручной оценки

### Принципы, которые из этого следуют
- **Объяснимость превыше точности** (см. `docs/task-rules.md` §5)
- Каждое правило ошибки должно быть объяснимо пользователю-любителю без анатомии
- Избегаем чёрных ящиков на прототипе

## Когда пересматривать
- Когда собран датасет → возможен переход к гибриду A+B
- Когда фидбек пользователя покажет, что rule-based советы «шаблонные» → кандидат на добавление LLM (C)
- Когда потребуется анализ серий подач (вариативность) — возможно ML на трендах
