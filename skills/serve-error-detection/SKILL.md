# Скилл: Rule-based детекция ошибок подачи

## Когда использовать

Перед любой задачей, которая:
- Реализует правила поиска ошибок в подаче
- Формулирует советы/фидбек для пользователя-любителя
- Определяет пороги биомеханики (углы, тайминги, расстояния)
- Работает со **слоями анализа** (1: простые советы, 2: метрики, 3: эталон)

## Принципы (из ADR-0002 и task-rules.md §5)

1. **Объяснимость превыше точности.** Каждое правило ошибки пользователь-любитель должен понять без анатомии.
2. **Tolerance-зоны, не точные пороги.** У любителей широкий разброс — используем диапазоны «нормы», а не точные значения.
3. **Каждый порог — именованная константа с источником.** Никаких магических чисел в коде.
4. **Не навреди.** Если правило сомневается → помечаем low-confidence, а не выдаём категоричный совет.

## Структура правила

Каждое правило ошибки описывается единым форматом:

```typescript
interface ErrorRule {
  id: string;                      // уникальный идентификатор
  phase: Phase;                    // к какой фазе относится
  layer: 1 | 2 | 3;                // слой глубины (1=простой совет)
  title: string;                   // короткое название (для UI)
  check: (ctx: PhaseContext) => Finding | null;
  // если возвращает null — ошибки нет / нельзя определить
}
```

```typescript
interface Finding {
  ruleId: string;
  severity: 'info' | 'warn' | 'error';  // насколько критично
  confidence: 'low' | 'medium' | 'high';
  advice: string;                  // текст совета для пользователя (без анатомии!)
  metric?: {                       // для Layer 2 — конкретное значение
    name: string;
    value: number;
    unit: string;
    referenceRange?: [number, number];
  };
}
```

## Кандидаты правил (из биомеханики)

Из `docs/biomechanics/serve-phases.md` — типичные ошибки любителей. Каждая — кандидат на правило.

> ⚠️ Конкретные пороги определяются эмпирически на тестовых подачах и заносятся в `src/constants/biomechanics.ts`. Здесь — **логика** правил.

### Trophy position

#### T1. Racket drop отсутствует
- **Логика:** в trophy ракеточная wrist выше головы, но **локоть не опущен** (ракетка не «падает» за спину).
- **Как измерить:** в trophy phase, проверяем что wrist ракеточной руки ниже elbow (ракетка опущена) — это racket drop.
- **Совет (Layer 1):** «В позиции „трофей" ракетка должна опуститься за спиной — это даёт ускорение на ударе. Попробуйте расслабить запястье и дать ракетке опуститься.»

#### T2. Слишком долго в trophy
- **Логика:** время от trophy event до начала acceleration больше нормы → игрок «зависает».
- **Как измерить:** длительность фазы trophy в кадрах, сравнить с типичной (напр. >N кадров при 30fps).
- **Совет:** «Пауза в позиции „трофей" отнимает энергию. Старайтесь плавно переходить от подброса к удару без остановки.»

#### T3. Тосс-рука падает слишком рано
- **Логика:** тоссовая wrist опускается **до** contact → игрок теряет «указатель» на мяч.
- **Как измерить:** высота тоссовой wrist на моменте contact; если существенно ниже её пика — рука упала.
- **Совет:** «После подброса оставьте тоссовую руку наверху чуть дольше — она помогает держать мяч в поле зрения и направление удара.»

### Toss

#### TO1. Подброс слишком далеко назад
- **Логика:** положение пика подброса мяча/тосс-руки позади тела (относительно hips) → игрок отклоняется назад при контакте.
- **Как измерить:** x-координата пика подброса относительно центра бёдер; если значительно позади — проблема.
- **Совет:** «Мяч подброшен слишком далеко назад. Старайтесь подбрасывать чуть вперёд и в сторону — так удар будет по линии и с весом.»

#### TO2. Подброс слишком низкий
- **Логика:** высота пика подброса (от release до peak) меньше нормы → игрок спешит с замахом.
- **Как измерить:** разница y-координат release и peak тосс-руки (как proxy для мяча).
- **Совет:** «Подброс слишком низкий — не хватает времени на полный замах. Подбрасывайте выше, чтобы успеть сделать полный круг ракеткой.»

### Contact

#### C1. Контакт слишком низко
- **Логика:** на contact высота wrist ракеточной руки существенно ниже максимума её вытяжения, либо локоть сильно согнут (<180° - tol).
- **Как измерить:** угол локтя в момент contact; высота wrist относительно плеча.
- **Совет:** «Удар происходит слишком низко — бьёте согнутой рукой. Тянитесь к мячу полностью выпрямленной рукой в верхней точке.»

#### C2. Контакт позади тела
- **Логика:** на contact мяч/точка удара позади тела (относительно forward direction).
- **Как измерить:** x-координата точки контакта относительно hips/shoulders; если позади → проблема.
- **Совед:** «Точка удара позади тела — мяч уходит в сетку или летит слишком длинно. Старайтесь бить по мячу чуть впереди себя.»

#### C3. Недостаточное сгибание коленей
- **Логика:** max knee flexion в trophy меньше нормы → игрок не использует ноги.
- **Как измерить:** минимальный угол колена (max flexion) в trophy phase.
- **Совет:** «Колени согнуты слабо — ноги не дают энергию удару. Сгибайте колени глубже в позиции „трофей", чтобы вытолкнуться вверх к мячу.»

### Follow-through

#### F1. Резкая остановка
- **Логика:** после contact ракетка не проходит через тело / движение резко обрывается.
- **Как измерить:** траектория ракеточной wrist после contact — должна пересечь среднюю линию тела и опуститься.
- **Совет:** «Движение ракетки обрывается после удара. Завершайте подачу — позвольте ракетке пройти через тело к противоположному боку. Это и мощность, и защита плеча.»

#### F2. Потеря баланса
- **Логика:** после подачи игрок существенно отклоняется / «заваливается».
- **Как измерить:** положение центра масс (hips) относительно стоп в конце follow-through; наклон.
- **Уровень уверенности:** обычно low — CV плохо отличает «заваливается» от «естественный шаг». Помечать info, не warn.

## Слои анализа

### Layer 1 — Простые советы (по умолчанию для всех)
- Вывод: текстовые советы из правил выше
- Сортировка по severity (error → warn → info)
- Без чисел, градусов, анатомии
- Это **основной фронтенд** прототипа

### Layer 2 — Точные метрики (опционально)
- К каждому совету (если применимо) — конкретное значение:
  - «Ваш угол сгиба колена: 12° (норма 20–35°)»
  - «Высота подброса: ~40см над головой (норма 50–80см)»
- Reference ranges из `src/constants/biomechanics.ts`
- Для любителя-продвинутого, которому интересны цифры

### Layer 3 — Сравнение с эталоном (опционально, частично в прототипе)
- Overlaid skeleton пользователя vs эталонное про-видео
- Требует:
  - Набор эталонных видео (1–3 на начало)
  - Нормализация скелетов (масштаб, ориентация, временная синхронизация по фазам)
- На прототипе: **минимальная версия** — показываем эталонный скелет в той же фазе рядом. Полный overlay-alignment — future.

## Пороги и калибровка

### Принцип
- Пороги собираем в `src/constants/biomechanics.ts`
- Каждый — с комментарием-источником:
  ```typescript
  // Chow et al. (2012), Table 3: intermediate players show knee flexion 20-35° at trophy
  // Calibrated 2026-06-20 on 5 test serves; tolerance widened by ±5° for amateur variability.
  export const KNEE_FLEXION_NORMAL_RANGE_DEG: [number, number] = [20, 35];
  ```

### Источники порогов
1. **Литература** — Chow et al., MDPI 2024, Frontiers 2024 (см. `docs/research/`)
2. **Практические** — OnCourtAI, APOPT метрики
3. **Эмпирические** — калибровка на тестовых подачах (preferred для прототипа)

### Tolerance-зоны
- Любительская подача **вариативна** — узкие пороги дадут много false positives
- Используем **диапазоны нормы** + severity по удалению от диапазона:
  - Внутри диапазона → no finding
  - Слегка вне → `info` / `warn`
  - Существенно вне → `error`

## Уверенность (confidence)

Каждое правило возвращает confidence. Факторы, понижающие confidence:
- Низкая visibility landmarks (см. cv-pose-estimation скилл)
- Короткая фаза (мало кадров для анализа)
- Тангенциальное движение к камере (2D-неточность)
- Тестовое видео низкого качества

**Правило:** low-confidence findings показываем, но мягко («Возможно, ...», «Не удалось точно определить, но ...»), не категорично.

## Тестирование правил

### Unit-тесты
Для каждого правила — синтетический `PhaseContext` с известными значениями:
```typescript
test('C3 flags insufficient knee bend', () => {
  const ctx = makeCtx({ kneeFlexionAtTrophy: 12 });  // ниже нормы
  expect(ruleC3.check(ctx)?.severity).toBe('warn');
});

test('C3 passes for good knee bend', () => {
  const ctx = makeCtx({ kneeFlexionAtTrophy: 28 });  // в норме
  expect(ruleC3.check(ctx)).toBeNull();
});
```

### Граничные случаи
- Значение ровно на пороге
- Low-visibility → правило должно вернуть low-confidence или null
- Пустая/короткая фаза

## Антипаттерны (НЕ делать)

1. ❌ Точный порог без tolerance-зоны → false positives
2. ❌ Совет с анатомическими терминами («уменьшите угол внутренней ротации плеча на 15°») → любитель не поймёт
3. ❌ Категоричный совет при low-confidence → вводит в заблуждение
4. ❌ Магическое число в теле правила → невозможно калибровать
5. ❌ Правило без unit-теста → регрессии при изменении порогов
6. ❌ ML/LLM для оценки → нарушает ADR-0002 на прототипе
